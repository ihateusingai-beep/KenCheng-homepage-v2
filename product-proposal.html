// 2 · AUDIENCE

邊個用？點用？

受眾分成 3 個 tier，每個嘅 workflow 同 pain point 唔同。Proposal 要 balance 邊個先做。

🎯 Tier 1 — Ken 自己（primary）

Owner / curator。每日用 1-3 次，desktop primary。

Pain points

  • Tool 死 link 要手動 check
  • Bookmark 過嘅 tool 散晒
  • 用緊嘅 tool 要 scroll 搵
  • 課堂 pack 每次由零拼

Wants

  • Health badge 即時見到 status
  • Quick launch（1-click 開 bookmarked）
  • Group launch（一齊開 5 個）

👥 Tier 2 — Colleagues

同校老師、SEN 老師。偶爾訪，share link 入嚟。

Pain points

  • 唔知邊個 tool 啱 SEN 學生
  • Featured vs 全部冇分別
  • 冇 lesson plan 框架

Wants

  • SEN 標籤（audience=student）
  • Featured CTA 一 click 即試
  • Class-pack download

🎒 Tier 3 — Students

偶爾訪（用 SEN math tool / 粵語朗讀）。手機 primary。

Pain points

  • Site 太花，搵唔到入口
  • Filter bar 7 chips 太多
  • Loading 慢（無 screenshot）

Wants

  • Direct deep link
  • Quick visual scan（screenshot）
  • Share button
💡 決策原則 Tier 1 嘅 pain 排最前（T1 = primary use case，desktop）。T2/T3 嘅改善唔好阻礙 T1。Mobile 唔使 single-primary（參考 v2 設計是 desktop-first）。
// 3 · STRATEGY

三條 principle 守住

Scope 收緊，避免過度 engineering。v2 嘅 success 係「ken 每日用得爽」，唔係「全部人愛」

① Static-only invariant

任何 feature 都唔可以加 backend。Bookmark、notes、recent 全部 localStorage。AI-assisted 嘅 call 用 client-side 嘅 redirect 到 claude.ai，唔 hosting model。

② Ken-first, not everyone-first

Site 主要 owner 自己用。Colleague 訪客 5-10% traffic。Design decision 嘅 tie-breaker 一律「Ken 用得爽」，唔係「generic 受眾用得爽」。

③ Refactor over add

每加一個新 feature 前，audit 已有嘅嘢可唔可以 reuse。Avoid feature creep — quality > quantity。Refactor done 先再 add。

// 7 · AUTOMATION

6 個自動化

減 Ken 手動維護成本。所有 build-time 或 nightly GitHub Action, 0 持續 server cost。

A1 — Nightly Link Checker

GitHub Action 凌晨 03:00 HKT 跑：fetch 63 URLs，response time 分級 (alive < 3s / slow 3-10s / dead > 10s or 4xx/5xx)，寫返 health + last_checked field 落 frontmatter。Auto-create GitHub Issue for newly dead tools。

# .github/workflows/nightly-link-check.yml
on:
  schedule: - cron: '0 19 * * *' # 03:00 HKT
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: node scripts/check-links.mjs
      - uses: peter-evans/create-commit@v6
P0 Effort M

A2 — OG Image Auto-gen

Build time 用 @vercel/og + satori 自動生成每個 tool page 嘅 OG image（1200×630, 顯示 tool name + cat + Ken 嘅 tagline）。Static public/og/<slug>.png。Falls back to site default OG if missing。

P0 Effort L

A3 — Card Screenshot via Playwright

Build time 對每個 tool URL 開 headless browser，截 1200×630 hero region，存 public/screenshots/<slug>.png。需要 network 訪問 external 站。Astro 5+ 有 astro:assets remote patterns 支援。

P1 Effort L

A4 — ✅ Shipped: RSS + Sitemap + Canonical

@astrojs/rss + @astrojs/sitemap。Extend: per-category feed /<cat>/rss.xml，方便訂閱者只 follow 數學。

✓ SHIPPED P1 Effort S

A5 — A11y Auto-test (axe-core)

CI 入 axe-core 跑 3 routes（/、/AI/、/數學/），filter `serious`+`critical`。3 routes 就夠 smoke。常見 violation：<span aria-label> 冇 role。Failing build = 不可 merge。

P1 Effort M

A6 — ✅ Shipped: CI Smoke + Lighthouse

// 11 · SUCCESS METRICS

點樣知 work？

由「感覺 work」升級做「measure work」。用 localStorage 收集 anonymous use data（opt-out default）。

📊 北極星指標

「Ken 每日 click 出去 tool 嘅次數」 — 透過 outbound link click 計 localStorage counter，每週 7+ = healthy，< 3 = stale。

  • Sprint 1-2 baseline: ~10/week (estimated)
  • Sprint 3-4 target: ~25/week (after F5/F8 add)
  • Q4 target: ~40/week (with class-pack)

🎯 Guardrail 指標

  • Site load LCP < 1.5s (Lighthouse mobile)
  • Site total dist < 250KB (Astro build output)
  • Zero 4xx/5xx on internal links (CI invariant)
  • Tool dead rate < 5% (A1 nightly tracker)
  • Lighthouse perf + a11y + SEO + best-practices ≥ 90
  • A11y axe-core critical = 0

📈 領先指標 (Leading)

  • Bookmark count per user (localStorage, opt-in)
  • ⌘K palette usage (open / open-and-pick ratio)
  • Featured CTA click rate (if measurable)
  • Reduce-motion toggle % (a11y 友善指標)

🚨 Counter-metrics (避免 metric gaming)

  • Tool count 太低 (under 50 = 唔 curated 夠)
  • Tool count 太高 (over 100 = 失 curated 性質)
  • Featured % over 30% = featured 冇特別
  • Health dead rate over 10% = link rot 失控
  • localStorage size > 1MB (per user) = 太貪心
